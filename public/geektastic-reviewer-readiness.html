Review approach

Separate confirmed behavior from likely risk

The review style is evidence-first: identify the intended behavior, inspect the implementation path, check untested or surprising defaults, and write feedback that another engineer can act on without guessing the reviewer's reasoning.

Assessment fit

What this packet is meant to demonstrate

SignalWhat to look for
CorrectnessDoes the implementation match the stated behavior, including defaults and error paths?
Edge casesAre null, undefined, empty input, nested data, async failure, and invalid state considered?
MaintainabilityIs the code easy to read, scoped to the problem, and consistent with local patterns?
Test evidenceDo the tests prove the behavior that matters, or only exercise the happy path?
Written feedbackCan the recipient understand severity, impact, and next action quickly?

Privacy boundary

Assessment-safe handling

This public packet does not include private assessment prompts, candidate code, platform screenshots, credentials, account data, payment details, contracts, or copied third-party material. Private review work should stay inside the platform that owns it.
