Review method

How I evaluate code

The reviewer workflow is intentionally simple: make the expected behavior explicit, identify risky inputs, add focused tests, compare the implementation against those checks, then explain the verdict and remaining risk.

1State the contract

Restate expected behavior in plain language before judging the implementation.

2Test the boundaries

Cover normal, empty, invalid, malformed, and unsafe inputs with focused checks.

3Explain the verdict

Give an accept, revise, or reject decision with evidence and residual risk.

Sample answer

Reviewer verdict shape

Verdict:
The solution should be revised.

Main issue:
It silently accepts unknown duration units and returns 0, which can hide bad input.

Evidence:
The existing behavior does not distinguish "5 parsecs" from a valid zero-duration input.

Suggested fix:
Reject unknown units with an explicit error and add tests for invalid units.

Residual risk:
The parser still needs a product decision for compound durations such as "1h 30m".

Public boundary

Safe to review without private context

The examples are public, original review artifacts. They do not include private source code, credentials, account exports, payment data, tax records, platform-restricted assessment content, customer data, private messages, cookies, or account screenshots.
