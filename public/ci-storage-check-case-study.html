Skip storage-layout checks for brand-new Solidity artifacts

A compact review page showing how to preserve storage-safety checks for modified upgradeable code while avoiding false CI failures for newly added contracts and libraries.

Problem

New files cannot have previous storage artifacts

The workflow was creating storage-check matrix entries from all changed Solidity files. That means a newly added contract or library could be sent to a storage comparison action even though no prior artifact exists for that file.

Patch

Build the matrix from modified files

Existing contracts and libraries still run through the storage check. Newly added files are excluded from the matrix because there is no prior storage layout to compare against.

if: steps.changed-contracts.outputs.contracts_modified_files != ''
env:
  CHANGED_CONTRACTS: ${{ steps.changed-contracts.outputs.contracts_modified_files }}

QA

Expected behavior by scenario

ScenarioExpected behavior
No storage updatesNo matching modified files, so no storage-check matrix is created.
Existing contract updated with no collisionThe modified contract remains in the matrix and is checked.
Existing contract updated with collisionThe modified contract remains in the matrix and the existing storage action can fail the workflow.
New contract or library addedThe new file is not in the modified-file matrix, so the workflow does not look for a missing baseline artifact.
