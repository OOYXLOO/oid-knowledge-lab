Context-aware contributor recommendations

A focused ranking improvement that combines semantic similarity with repository context, so same-project experience can outrank unrelated high-similarity matches.

Problem

Semantic similarity alone can recommend the wrong contributor

A highly similar issue from a different organization may be less useful than a slightly less similar issue from the same repository. The recommendation should preserve broad semantic search while allowing project context to influence the final order.

Solution

Apply lightweight repository-context weighting

Match contextMultiplierReason
Same repository100%Most likely to reflect relevant local knowledge.
Same owner or organization75%Still likely to share conventions, stack, or review expectations.
Different owner or organization50%Useful semantic signal, but weaker project fit.

Regression Test

Same-repository experience ranks first

The test compares a 98% cross-organization match with an 82% same-repository match. After context weighting, the same-repository contributor remains at 82%, while the cross-organization contributor drops to 49%.

npx --yes bun test tests/main.test.ts --test-name-pattern "same repository context"

Takeaway

Use business context after broad retrieval

Embedding search is a strong retrieval signal, but final ranking should reflect how the result will be used. A simple context multiplier improved recommendation relevance without adding a new model, changing storage, or narrowing retrieval too early.
