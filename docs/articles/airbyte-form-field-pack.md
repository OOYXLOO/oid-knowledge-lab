# Airbyte form field pack

Use this packet when preparing the Airbyte Write for the Community Typeform. It is a planning aid for a fresh human-written submission, not a final article draft.

## Current public rules checked

- Airbyte invites data engineers, analytics engineers, and software developers to write data or AI knowledge articles.
- Accepted writers can earn between USD 300 and USD 500 per article, including a performance bonus.
- Airbyte asks for either data insights or tutorials that teach practitioners how to use Airbyte to build data and AI solutions.
- The process starts by submitting a draft through Typeform.
- The draft should align with an unassigned topic from Airbyte's GitHub project and be shared in Google Docs or a similar editor.
- Airbyte says AI-written drafts are not accepted.
- If selected, Airbyte may assign the GitHub item and continue through collaborative editing.
- Payment is processed after publication and performance assessment, using Deel.

## Safer submission strategy

The current OID pipeline topic should be submitted as a draft plan / outline only after it is rewritten by the author in their own words.

If the form requires a draft link, use a Google Doc or similar editor that contains:

- a human-written introduction;
- a human-written outline;
- one short original section;
- proof links to implementation artifacts;
- a clear note that the linked worksheet is planning material, not the final draft.

## Field-ready answers

### Working title

```text
Build an AI-ready OID knowledge pipeline with Airbyte, Postgres, and full-text search
```

### Short topic summary

```text
A hands-on Airbyte tutorial about turning a public registry-style knowledge source into a refreshable, queryable Postgres knowledge layer for developer tools and AI context workflows.
```

### Why this fits Airbyte

```text
The article focuses on data movement, source boundaries, sync design, schema modeling, change detection, and queryable AI context. It is a practical tutorial rather than a generic overview, and it maps naturally to Airbyte source ingestion plus a Postgres destination.
```

### Proposed reader

```text
Data engineers, analytics engineers, developer-tool builders, and AI application builders who need a repeatable way to turn public technical registries into reliable context stores.
```

### Proposed outline

```text
1. Why registry-style knowledge is useful but hard to query directly.
2. Define the safe source boundary and publication boundary.
3. Choose an Airbyte ingestion path for public records.
4. Model OID nodes, parent-child relationships, source URLs, and metadata in Postgres.
5. Track changes with stable keys, timestamps, and content hashes.
6. Add full-text search and subtree queries.
7. Expose the data for developer tools or AI context workflows.
8. Validate the pipeline with manifests, commands, and derived proof artifacts.
```
