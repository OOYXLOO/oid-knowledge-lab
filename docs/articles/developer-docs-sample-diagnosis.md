# Developer Docs Sample Diagnosis

This sample shows the kind of small, concrete documentation diagnosis that can start a paid README, quickstart, or DevRel trial.

## Scenario

A developer-tool project has a working repository and a public demo, but a new reviewer still needs too much time to answer the first five questions:

- What does it do?
- Who is it for?
- What can I run today?
- What proof is live?
- What is still a prototype, dry run, mock, or future plan?

## First-minute diagnosis

The common problem is not missing polish. It is missing evaluation order. A technical reviewer should not have to read the architecture, source tree, release notes, and marketing copy before understanding the safe first path.

## Before

Typical unclear opening:

```text
This project is an AI-powered platform for managing complex workflows with integrations, evidence, automation, and intelligent agents.
```

Problems:

- too broad to evaluate quickly
- no specific user or workflow
- no first action
- no proof boundary
- no distinction between live behavior and planned integrations

## After

Clearer opening:

```text
This project helps small developer-tool teams prepare a launch review packet from public or sanitized materials. It maps the first-minute reviewer path, rewrites the README opening and quickstart, and separates live proof from dry-run proof, mocks, and future claims.
```

Why this works better:

- names the audience
- names the workflow
- names the deliverables
- avoids unsupported platform claims
- makes the trust boundary visible

## Suggested first trial

The first trial should produce:

- a first-minute developer path map
- a rewritten README opening section
- a revised quickstart or first-run path
- a proof and claims checklist
- a short backlog for follow-up docs or launch work

## Acceptance signal

A new reviewer can understand the project, strongest claim, proof state, first action, and missing pieces without needing a call.
