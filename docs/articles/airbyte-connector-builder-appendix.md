# Airbyte Connector Builder appendix: safe registry evidence source

This appendix supports the article proposal "Build a Safe Registry Evidence Dashboard from Public and Local Data." It shows how the local OID Knowledge Lab workflow can be translated into Airbyte product concepts without requiring private credentials, private customer inventories, or copied third-party page bodies.

The goal is not to claim that the public sample is already a production Airbyte connector. The goal is to give an editor or reviewer a concrete adaptation path from the working local implementation to an Airbyte-oriented tutorial.

## Tutorial mapping

| Workflow concern | Local implementation | Airbyte-oriented article section |
| --- | --- | --- |
| Public source | Public registry sitemap and public metadata pointers | Connector Builder or Low-code CDK source prototype |
| Safe local source | Sanitized CSV asset inventory | File source with only approved, non-sensitive columns |
| Normalize | Node.js parsing and OID validation | Post-sync model, dbt step, or destination-side transform |
| Review output | JSON, Markdown, CSV, and static dashboard | Local JSON destination or warehouse destination feeding review artifacts |
| Publish boundary | `guard-publishable` and dataset manifest | CI release guard before generated public artifacts |
| Reviewer proof | Public site, raw docs, reproducible commands | Article verification section and copyable checklist |

## Minimal public-source stream shape

A Connector Builder prototype for the public source can start with a tiny stream contract:

```json
{
  "stream": "public_registry_entries",
  "primary_key": ["oid"],
  "schema": {
    "type": "object",
    "required": ["oid", "source_url", "source_kind"],
    "properties": {
      "oid": { "type": "string" },
      "source_url": { "type": "string" },
      "source_kind": { "type": "string" },
      "public_label": { "type": ["string", "null"] },
      "last_observed_at": { "type": ["string", "null"], "format": "date-time" }
    }
  }
}
```

The public tutorial should keep this stream intentionally narrow. It should not forward arbitrary page bodies, private review notes, credentials, account exports, or client-specific fields.

## Sanitized inventory stream shape

The local inventory should be treated as a separate source. In an Airbyte article, that can be modeled as a File source after the inventory has already been sanitized:

```csv
asset,oid
router-core,1.3.6.1.4.1.9.9.41
sha256-policy,2.16.840.1.101.3.4.2.1
unknown-enterprise,1.3.6.1.4.1.999999.1
bad-row,not-an-oid
```

The article can call out that the public example accepts only a safe asset label, an OID value, and a row index generated by the parser.

It should explicitly reject credentials, account exports, raw logs, customer names, contact-level data, payment data, internal tickets, and copied third-party page bodies.

## Destination shape

A small Local JSON destination or warehouse table can use this review-oriented output shape:

```json
{
  "asset": "router-core",
  "oid": "1.3.6.1.4.1.9.9.41",
  "status": "matched_public_evidence",
  "risk": "low",
  "source_trust": "public_registry_pointer",
  "review_action": "keep",
  "public_evidence_url": "https://example.invalid/public-registry-pointer"
}
```

The generated dashboard can then read derived rows instead of raw source inputs.

## Validation checklist

The Airbyte-oriented tutorial should include a verification section with commands and acceptance checks:

```text
npm run check
npm test
npm run audit:assets
npm run coverage:oid
npm run build:site
npm run guard:publishable
```

The important checks are:

1. Invalid OIDs are detected before publishing.
2. Unknown private enterprise numbers remain review items, not silent successes.
3. Public artifacts contain derived review rows, not private source exports.
4. The dataset manifest records generated artifact counts and expected paths.
5. The publish guard blocks disallowed local evidence files from becoming public.

## Editorial angle

The final Airbyte article can be written as a practical pattern:

> Build a small public-source connector concept, combine it with a sanitized file source, normalize the rows, and publish a safe review dashboard with a CI guard.

That keeps the article aligned with Airbyte readers while preserving the public/private source boundary that makes the sample safe to inspect.
