# Draft.dev writer profile one-pager

This one-pager supports a Draft.dev writer-network application. It is not a single article submission or finished client draft.

## Positioning

Implementation-backed developer writing for data workflows, observability handoffs, release guards, API debugging, generated documentation systems, and static review artifacts.

## Best-fit assignments

I am strongest when an editor or client has a technical product, API, workflow, or developer-tool brief and needs a practical article that readers can reproduce. The ideal assignment has a concrete reader outcome, a product or workflow to demonstrate, and enough technical detail to verify the example before writing.

## Delivery style

1. Clarify the reader outcome and client/editor constraints.
2. Build or verify a small working example.
3. Produce an outline with prerequisites, code path, expected output, and acceptance checks.
4. Write the draft in a practical developer voice.
5. Revise against editor or SME notes without weakening technical accuracy.

## Topic lanes

- Developer tooling and release safety.
- Observability and production debugging.
- API, webhook, queue, and SaaS integration troubleshooting.
- Static evidence dashboards and generated documentation.
- Data quality review surfaces with safe publication boundaries.

## Boundary

The public samples are original developer-education materials built around safe derived artifacts. They do not publish credentials, private customer inventories, payment data, account exports, contact-level registry exports, API tokens, cookies, raw third-party page-body mirrors, or local-only crawl output.

The application should present this as a writer profile and sample set, not as an AI-generated full draft submission.
