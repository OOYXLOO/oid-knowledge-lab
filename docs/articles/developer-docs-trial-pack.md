# Developer Docs Trial Pack

This pack is a neutral review surface for developer-tool teams that need a small, measurable documentation trial before committing to a larger DevRel or documentation engagement.

## Trial shape

The first trial should be narrow enough to review in one week:

- audit the README, quickstart, examples, and launch copy
- identify the first-minute developer path
- rewrite the top-level positioning and quickstart section
- separate live proof, dry-run proof, mocks, and future claims
- leave a short content backlog for the next two or three weeks

## Why this is useful

Developer-tool projects often have enough implementation work to be interesting, but the first reviewer still has to answer basic questions:

- What does this do?
- Who is it for?
- What can I run today?
- Which claims are already proven?
- Which claims should stay out of launch copy until they have evidence?

This trial turns those questions into a concrete handoff instead of a vague content strategy conversation.

## Safe input boundary

Useful inputs:

- public repository or documentation link
- sanitized README, quickstart, or landing page copy
- existing screenshots, demos, or release notes
- one paragraph about the intended audience

Excluded inputs:

- passwords, API keys, cookies, or private account exports
- customer data or private production logs
- payment, KYC, tax, or billing material
- unsupported claims that cannot be shown publicly

## Deliverables

The recommended first trial produces:

- a first-minute developer path map
- a rewritten README opening section
- a revised quickstart or first-run path
- a proof and claims checklist
- a small content backlog for follow-up work

## Acceptance criteria

The trial is complete when a new technical reviewer can open the repository or landing page and understand the product, first use case, run path, evidence boundary, and next action without needing a call.
