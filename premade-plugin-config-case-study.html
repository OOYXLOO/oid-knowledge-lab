Product configuration case study

Turn low-level plugin defaults into named premade configurations

A small installer UX change that makes partner onboarding less ambiguous: show a named standard configuration first, while preserving custom editing for installed plugins.

Configuration shape

Make the product promise explicit

The standard configuration now has a name, description, and version. That gives the UI a product-level label instead of exposing an implementation-level template name.

interface PremadeConfig {
  metadata: { name: string; description: string; version: string };
  plugins: Record<string, { yamlConfig: string }>;
}

Review frame

Default path without removing control

The primary selector can show `UbiquityOS Standard Config` and `Custom`. New partners get a clear recommended path; experienced teams can still edit installed plugin configuration directly.
