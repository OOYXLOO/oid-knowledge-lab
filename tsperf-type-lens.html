Problem

Type complexity is hard to inspect during review

Large TypeScript libraries can hide expensive conditional, mapped, recursive, or template-literal type declarations behind small aliases. TSPerf Type Lens makes those hotspots visible near the source before they become editor latency or CI compile-time surprises.

Current implementation

Signals collected per declaration

SignalPurpose
Checker timingMeasures declared-type resolution, type string expansion, and total checker time for a declaration.
Structural depthRanks nested helper shapes that often indicate hard-to-review type computation.
Syntax driversCounts conditional, mapped, infer, indexed-access, union, intersection, keyof, and template-literal features.
Reference footprintTracks identifier references and unique references to highlight wide helper dependencies.
RecommendationsSuggests trace capture, recursion guards, branch splitting, key precomputation, or alias simplification.

The current source is a research prototype, not a claim that all TypeScript editor latency is explained by these heuristics. The next stronger validation step is to compare hotspots against tsserver traces from larger public projects.

Next research step

Move from heuristic ranking to trace-backed evidence

The useful next increment is to collect a small public benchmark set, run TSPerf Type Lens, capture TypeScript server traces for the top hotspots, and publish a comparison table that shows where the heuristic matches or misses real editor cost.
